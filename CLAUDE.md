**https://claude.ai/code/artifact/9944c8ca-31a6-4695-b6b2-c11b8e58aeb7**

## Qué es esto

Backend (NestJS + PostgreSQL) de **SmartBox**, un SaaS de gestión para gimnasios: multi-tenant, membresías con facturación recurrente, reservas. El control físico de cápsulas/lockers vía IoT (la visión original del producto) es deliberadamente la **última** release del roadmap ("SmartBox IoT"), no la primera — no lo prioricés a menos que el documento oficial diga lo contrario.

## Decisiones de alcance ya tomadas para v1.0 (no las reabras sin avisar)

- ~~Un solo plan de membresía por gimnasio al arrancar~~ — **superado por `E6-04` (2026-07-20)**, ver la sección dedicada más abajo. Sigue sin haber descuentos ni cupones, eso quedó explícitamente fuera del alcance de `E6-04`.
- **Proveedor de pagos: Mercado Pago, no Stripe** (cambio de proveedor, 2026-07-17) — Stripe no permite abrir cuenta desde Ecuador (Ecuador no está en su lista de países soportados). Se evaluaron PayPal y PayPhone como alternativas; se eligió Mercado Pago por tener una API de suscripciones recurrentes (`PreApprovalPlan`/`PreApproval`) lista para usar y soporte real para Ecuador. SDK: paquete npm `mercadopago` (v3, cliente `MercadoPagoConfig`). **Matiz encontrado el `2026-07-23`**: "soporte real para Ecuador" es cierto para cuentas de vendedor comunes, pero no para crear una Aplicación/Marketplace (exige empresa en Argentina) — ver el bullet de conexión más abajo, se resolvió sin cambiar de proveedor.
- **Modelo Marketplace: cada gimnasio cobra en su propia cuenta de Mercado Pago, no SmartBox** (decisión de negocio, 2026-07-17) — la plata de los socios va directo al gimnasio; SmartBox nunca la toca ni la redistribuye (evita compliance de manejo de fondos de terceros). Sigue siendo así — solo cambió *cómo* se conecta la cuenta, ver `2026-07-23` más abajo.
  - `Gym` guarda `mercadoPagoUserId`, `mercadoPagoAccessToken` (`select: false`, igual que `User.password`) y, desde `2026-07-23`, `mercadoPagoWebhookSecret` (también `select: false`).
  - ~~El ADMIN conecta su cuenta vía OAuth: `GET /gyms/:id/mercadopago/connect`...~~ — **superado, `2026-07-23`**: Mercado Pago exige una empresa registrada en Argentina para crear la Aplicación que ese flujo OAuth necesitaba, y Ecuador no está soportado para ese producto. Ver la sección dedicada más abajo — ahora el gimnasio pega su propio access token, sin Aplicación de por medio.
  - `PlansService`/`MembershipsService` **nunca** llaman a Mercado Pago con un token global — siempre resuelven el `access_token` del gimnasio dueño vía `GymsService.getMercadoPagoAccessToken(gymId)` y arman un cliente por-request con `MercadoPagoService.clientFor(accessToken)`. Si un gimnasio no conectó su cuenta, crear un `Plan` o suscribirse falla con 400 explícito.
  - ~~Sin refresh automático de tokens todavía (gap conocido)~~ — **ya no aplica, `2026-07-23`**: un access token pegado a mano no tiene un flujo de refresh OAuth que implementar; si el gimnasio lo revoca, simplemente vuelve a pegar uno nuevo generado desde su cuenta.
- **Webhook de Mercado Pago (E2-03), implementado 2026-07-17, secreto por-gimnasio desde `2026-07-23`** — `POST /memberships/webhook/mercadopago` (público, sin JWT):
  - Firma verificada con `WebhookSignatureValidator` del SDK oficial (`import { WebhookSignatureValidator } from 'mercadopago'`) — no reinventar el HMAC a mano, el SDK ya lo resuelve. ~~Contra `MERCADOPAGO_WEBHOOK_SECRET`~~ — ya no hay un secreto global, ver `2026-07-23` más abajo: se resuelve primero a qué gimnasio pertenece la notificación y se verifica contra el secreto de *ese* gimnasio.
  - Idempotencia por `notification.id`: `INSERT` en `processed_webhook_events` (PK = id) y capturar el error de constraint única como señal de "ya procesado" — nunca "leer y después insertar" (deja una carrera entre notificaciones concurrentes).
  - El body de la notificación **nunca** es la fuente de verdad — solo dispara una consulta a la API real (`client.subscriptions.get({id: data.id})`) antes de tocar la `Membership`.
  - Maneja dos topics, distinguidos por `payload.type`: `subscription_preapproval`/`preapproval` (alta/cancelación de la suscripción, aceptados ambos de forma defensiva porque el string exacto todavía no está confirmado contra tráfico real de Mercado Pago) y `subscription_authorized_payment` (cobro recurrente/dunning, `E2-05` — ver más abajo). El topic genérico `payment` (pagos únicos, no recurrentes) se ignora a propósito.
- **Conexión de Mercado Pago: de OAuth con Aplicación a access token pegado a mano, 2026-07-23** — al intentar crear la Aplicación real de Mercado Pago que el flujo OAuth (`GET /gyms/:id/mercadopago/connect` → redirect → `GET /mercadopago/oauth/callback`) necesitaba, Mercado Pago exigió una empresa registrada en **Argentina** — Ecuador no está soportado para ese producto específico (confirmado que sí lo está para cuentas de vendedor comunes, la asunción original de "soporte real para Ecuador" de la decisión del `2026-07-17` de arriba resultó incompleta). Se evaluó cambiar de proveedor (Paddle, Kushki) antes de resolverlo así — descartado: Paddle es Merchant of Record (rompe el modelo Marketplace) y Kushki no tiene onboarding self-serve para su feature de split payments ni preserva la dirección del flujo de dinero (el "comisionado" en su modelo recibe una comisión chica, no el pago completo).
  - **Nuevo flujo**: el gimnasio genera su propio access token directamente desde su propia cuenta de Mercado Pago (Tu negocio → Configuración → Credenciales — sin crear ninguna Aplicación) y lo pega junto con el secreto de firma de su propio webhook en `PUT /gyms/:id/mercadopago/credentials` (reemplaza al viejo `GET .../connect`; `SUPER_ADMIN`/`ADMIN` propio). El backend valida el token contra `GET /users/me` (`MercadoPagoService.verifyAccessToken()`, usa el cliente `User` del SDK) antes de guardar nada — si es inválido/revocado, 400 explícito sin dejar al gimnasio "conectado" a medias.
  - `MercadoPagoOauthController` se eliminó por completo (ya no hay callback). `MercadoPagoService` perdió el cliente de plataforma (`OAuth`, `getAuthorizationUrl()`, `exchangeCodeForTokens()`) — ya no hace falta ninguna credencial a nivel de toda la app. `GymsService` perdió `startMercadoPagoConnect()`/`completeMercadoPagoConnect()` (y con ellos, la dependencia de `TokenService`, que sigue en uso para reset de contraseña/cambio de email) a favor de `connectMercadoPago(id, accessToken, webhookSecret)`.
  - **Consecuencia real, no obvia al principio**: sin una Aplicación centralizada tampoco hay un secreto de firma único para toda la plataforma — cada gimnasio configura su propio webhook en su propia cuenta y Mercado Pago le da su propio secreto. `Gym.mercadoPagoWebhookSecret` (nuevo, `select: false`) guarda ese secreto por-gimnasio; `GymsService.getMercadoPagoWebhookSecret(gymId)` lo expone con el mismo patrón que `getMercadoPagoAccessToken()`. Esto obligó a **reordenar** `MembershipsService.handleWebhook()`: antes verificaba la firma primero (con el secreto global) y recién después resolvía a qué gimnasio pertenecía la notificación; ahora resuelve el gimnasio primero (vía `payload.user_id`, una simple lectura, sin mutar nada) y verifica la firma contra el secreto de *ese* gimnasio específico — el insert de idempotencia (`processed_webhook_events`) sigue ocurriendo únicamente después de una verificación exitosa, así que la garantía de "nunca mutar antes de verificar" se mantiene igual que antes, solo cambió el orden de la parte de solo-lectura.
  - Se eliminaron las 5 variables de entorno `MERCADOPAGO_ACCESS_TOKEN`/`CLIENT_ID`/`CLIENT_SECRET`/`REDIRECT_URI`/`WEBHOOK_SECRET` (`env.validation.ts`, `.env.example`) — ninguna se lee más en el código, confirmado con `grep -rn "MERCADOPAGO_" src`. Esto de paso resuelve la fricción real que hubo desplegando a Railway con placeholders para estas 5 variables.
  - Migración `ReplaceMercadoPagoOauthWithWebhookSecret`: elimina `mercadopago_refresh_token`/`mercadopago_token_expires_at`/`mercadopago_oauth_state`/`mercadopago_oauth_state_expires_at` de `gyms`, agrega `mercadopago_webhook_secret`. Corrida contra Postgres local y verificada con `\d gyms` — las 4 columnas viejas ya no existen, la nueva sí.
  - Verificado en vivo end-to-end contra el servidor y la base de datos reales (fixture temporal por `POST /auth/signup-gym`, borrado después): el formulario nuevo de `smartbox-admin` (`MercadoPagoConnectionCard`) llama a `PUT /gyms/:id/mercadopago/credentials` con un token inválido a propósito, el backend efectivamente golpea la API real de Mercado Pago (`GET /users/me`) y devuelve 400 con `"El access token no es válido o fue revocado..."`, que el formulario muestra tal cual — confirma que todo el camino (formulario → endpoint → validación real contra Mercado Pago → error mostrado) funciona. No se pudo probar el camino de éxito (token real) en esta sesión — el primer gimnasio real que conecte su cuenta debería confirmarlo. 166 tests backend / build / lint limpios en ambos repos.
  - Mapeo de estado (eventos `subscription_preapproval`): `authorized` sin `Membership` previa → la crea (`active`, resolviendo el `Plan` vía `external_reference` → `User.gym`, no vía un campo `preapproval_plan_id` en la respuesta porque no está confirmado que ese campo venga en el `GET`); `authorized` con `Membership` existente → actualiza `currentPeriodEnd`; `cancelled` → marca la `Membership` como `cancelled` si existía.
- **Scoping de billing (Epic 2), cerrado 2026-07-16, adaptado a Mercado Pago el 2026-07-17** — ver el detalle en el documento oficial (§00, "6b · Resultado de la sesión de scoping de billing"):
  - Trial de **14 días**: se define una sola vez en el `PreApprovalPlan` (`auto_recurring.free_trial: {frequency: 14, frequency_type: 'days'}`), no por suscripción — todo CLIENT que se suscribe al plan de un gym hereda el mismo trial.
  - Alta de membresía **únicamente self-service con tarjeta**: `POST /memberships/subscribe` crea un `PreApproval` con `status: 'pending'` (sin `card_token_id`) y devuelve el `init_point` hosted de Mercado Pago para que el socio cargue la tarjeta ahí — cero UI de pago propia. Nada de alta manual/offline por ADMIN en v1.0.
  - **Sin reembolsos automáticos ni endpoint propio** — casos excepcionales se resuelven a mano desde el panel de Mercado Pago.
  - **Cancelación "hasta fin del período pagado" (E2-04), implementado 2026-07-17** — a diferencia de Stripe, la API de Mercado Pago no tiene un flag nativo equivalente a `cancel_at_period_end` (cancelar un `PreApproval` lo corta de inmediato), así que se simula a nivel de aplicación:
    - `POST /memberships/:id/cancel` (dueño de la membresía, o `ADMIN`/`SUPER_ADMIN` del gimnasio) solo marca `Membership.cancelAtPeriodEnd=true` — no toca Mercado Pago todavía, el socio sigue con `status: 'active'`.
    - `MembershipsService.processScheduledCancellations()`, con `@Cron(CronExpression.EVERY_DAY_AT_MIDNIGHT)` (registrado vía `ScheduleModule.forRoot()` en `AppModule`), busca `Membership`s con `cancelAtPeriodEnd=true` y `currentPeriodEnd` vencido, y recién ahí cancela el `PreApproval` real (`client.subscriptions.update({id, body: {status: 'cancelled'}})`) y marca `status: 'cancelled'` localmente.
    - Si la cancelación remota falla, `cancelAtPeriodEnd` queda en `true` sin tocar `status` — se reintenta en el próximo barrido en vez de perder el pedido silenciosamente.
  - **Dunning (E2-04→E2-05), implementado 2026-07-17** — confirmado por documentación oficial de Mercado Pago: los cobros recurrentes reportan su resultado por el topic `subscription_authorized_payment` (distinto del `payment` genérico de pagos únicos, y distinto de `subscription_preapproval`). En `MembershipsService.handleWebhook()`, un evento de este topic consulta el `Payment` real (`client.payments.get({id: data.id})`, `MercadoPagoService.clientFor()` ahora también expone `payments: Payment`) y llama a `syncMembershipFromPayment()`:
    - `status: 'rejected'` con `Membership.status === 'active'` → pasa a `past_due` (el socio **no** pierde el acceso todavía, por la sesión de scoping de billing).
    - `status: 'approved'` con `Membership.status === 'past_due'` → vuelve a `active`.
    - Correlación pago→socio vía `payment.external_reference` contra `Membership.userId` — **asunción sin confirmar contra tráfico real todavía**: se asume que Mercado Pago copia el `external_reference` del `PreApproval` original a los `Payment`s recurrentes que genera (no hay un campo `preapproval_id`/`subscription_id` explícito en el recurso `Payment` del SDK). Verificar en sandbox/producción antes de confiar en esto para dunning de alto volumen.
    - **La cadencia/cantidad de reintentos automáticos de Mercado Pago sigue sin documentación pública** — deliberadamente no se replica acá; el sistema solo reacciona a los eventos que Mercado Pago decida mandar. Si Mercado Pago termina cancelando el `PreApproval` tras agotar sus propios reintentos, el evento `subscription_preapproval`→`cancelled` (ya manejado) se encarga de reflejarlo.
    - Sin tests contra tráfico real de Mercado Pago (solo unit tests con firma HMAC real computada a mano y un smoke test manual con firma real pero payload sintético, ya que no hay credenciales de sandbox con pagos reales todavía) — si en producción `external_reference` no llega como se asume, dunning queda silenciosamente sin efecto (no falla, pero tampoco actualiza el estado). Vigilar logs (`Logger` de `MembershipsService`) al conectar el primer gimnasio real.
  - **Registro interno de facturas (E2-06), implementado 2026-07-17** — entidad `Invoice` (tabla `invoices`, `membershipId`, `amountCents`, `status`, `mercadoPagoPaymentId` único, `paidAt`). Se puebla exclusivamente desde `MembershipsService.recordInvoice()`, llamado dentro de `syncMembershipFromPayment()` (mismo evento de webhook que dispara el dunning de `E2-05`) — nunca hay alta manual ni endpoint dedicado en v1.0, es la fuente de datos para reportes de Epic 4.
    - Upsert por `mercadoPagoPaymentId`: si ya existe una factura para ese `Payment` (Mercado Pago puede notificar el mismo pago más de una vez, p. ej. creado y luego actualizado a su estado final), se actualiza `status`/`amountCents`/`paidAt` en vez de insertar una fila nueva.
    - `amountCents` se calcula desde `payment.transaction_amount` (float) con `Math.round(x * 100)`, siguiendo la misma convención de centavos que `Plan.priceCents`. `paidAt` viene de `payment.date_approved` (`null` si el pago no fue aprobado).
    - `status` guarda el string crudo del `Payment` de Mercado Pago (no un union acotado como `Membership.status`) — es un registro pasivo, no la fuente de la transición de estado de la `Membership`.

  - **Portal de gestión de plan/tarjeta/facturas del socio (E2-07), resuelto 2026-07-17** — investigado antes de programar, por el gap explícito que dejó la migración a Mercado Pago: no existe un endpoint tipo `stripe.billingPortal.sessions.create()` que genere un link hosted por-comercio. Lo que sí existe es la propia cuenta de Mercado Pago del socio (mercadopago.com → Tu perfil → Suscripciones/Débitos automáticos), donde ve y puede cancelar cualquier suscripción de cualquier comercio — no es algo que SmartBox integre ni pueda deep-linkear (no hay API para eso). `PUT /preapproval/{id}` sí acepta `card_token_id` para cambiar el medio de pago, pero usarlo exigiría un formulario de tarjeta propio (Brick/CardForm), lo que reabre "cero UI de pago propia" — se decidió explícitamente no hacerlo en v1.0.
    - Alcance elegido: **solo endpoints de lectura**, sin UI de pago ni link a ningún portal. `GET /memberships/me` (`CLIENT`) devuelve la membresía más reciente del solicitante (con el `Plan`); `GET /memberships/me/invoices` (`CLIENT`) devuelve su historial de `Invoice` (de todas sus membresías, no solo la vigente). Ambos scopeados implícitamente por `requester.id` — sin parámetro `:id`, no hay superficie de cruce entre socios que testear con 403.

- **Epic 3 (Reservas), implementado 2026-07-17** — el backlog oficial (`E3-01`–`E3-05`) tenía el modelo `ClassOrResource`/`Reservation` marcado explícitamente como "punto de partida, a validar", sin definir cómo se generan los horarios reservables de una clase ni resolver la ambigüedad entre los 6 estados sugeridos para `Reservation` (algunos pensados para el check-in físico de Epic 8). Se resolvieron ambos puntos antes de programar:
  - **Modelo de horarios: plantilla recurrente con generación de ocurrencias** (decisión de negocio, 2026-07-17) — `ClassOrResource` (tabla `classes`) es una plantilla semanal (`gymId`, `name`, `capacity`, `dayOfWeek` 0-6, `startTime` `HH:mm`, `durationMinutes`), no una fila por ocurrencia. Una clase que se dicta varios días por semana se modela con varias filas (mismo `name`, distinto `dayOfWeek`). Las ocurrencias reservables **no se materializan en una tabla aparte** — se derivan en el momento a partir del patrón, con funciones puras en `src/modules/classes/occurrence.util.ts` (`computeOccurrences` para `GET /classes/:id/availability`, `isValidOccurrence` para validar el `startAt` que manda el socio al reservar). Sin soporte de timezone por gimnasio todavía — el cálculo usa la hora del servidor, igual que el resto del dominio de billing (`Gym.timezone` existe pero no se usa acá).
  - **Modelo de estados de `Reservation`: simplificado** (decisión de negocio, 2026-07-17) — solo `confirmed` / `cancelled` / `expired`, sin `pending`/`active`/`finished` (esos últimos dos estaban pensados para el check-in físico de un dispositivo IoT, Epic 8, que todavía no tiene contrato de API diseñado). Una reserva nace `confirmed`; pasa a `cancelled` si el socio/`ADMIN`/`SUPER_ADMIN` la cancela (`POST /reservations/:id/cancel`, mismo patrón que `MembershipsService.requestCancellation`); pasa a `expired` vía `ReservationsService.expireReservations()`, `@Cron(CronExpression.EVERY_HOUR)`, `UPDATE` masivo sin llamada externa de por medio (a diferencia del cron de cancelación de `Membership`, acá no hay ningún proveedor externo que avisar).
  - **Validaciones de `POST /reservations` (E3-03), en este orden**: la clase pertenece al gimnasio del solicitante (403, no 404, si no); la membresía del solicitante está `active` (400 con mensaje explícito, no un error genérico — así lo pide HU-03 del documento oficial); el `startAt` pedido coincide exactamente con una ocurrencia real del patrón (`isValidOccurrence`); hay cupo (`capacity` menos reservas `confirmed` para esa clase+horario); el socio no tiene otra reserva `confirmed` que se superponga en el tiempo (`startAt < endAt_existente AND endAt > startAt_existente` contra todas sus reservas, no solo las de la misma clase — no podés estar en dos clases a la vez).
  - Gestión de `ClassOrResource` (alta): restringida a `SUPER_ADMIN`/`ADMIN`, mismo patrón que `Plan` (forzado al propio gym para `ADMIN`) — el backlog oficial no definía este endpoint explícitamente (solo mencionaba el de disponibilidad), se agregó por ser un prerrequisito obvio y mecánico para que `E3-01` funcione, sin necesidad de una sesión de scoping aparte.
  - Verificado en vivo contra la base de datos y el servidor reales: alta de clase, disponibilidad con cupo correcto, reserva, rechazo por solapamiento, cupo decreciente tras reservar y creciente tras cancelar, rechazo 400 con mensaje explícito sin membresía activa, y 403 (no 404) cruzando gimnasios.

Solo reporta excepciones no controladas o 5xx — los `HttpException` esperados (`NotFoundException`, `BadRequestException`, etc.) siguen devolviendo su JSON de siempre sin llegar a Sentry, verificado en vivo (404/401/400 probados contra el server real, formato de respuesta idéntico a antes de este cambio).

## Auditoría de seguridad y consistencia, 2026-07-21 (post-v1.0, con Epic 6 en progreso)

Auditoría completa del código (guards/roles/scoping en todos los controllers, entidades, migraciones) pedida explícitamente por el usuario ("puedes analizar el proyecto completo... editar arquitectura roles etc"). Encontró una vulnerabilidad crítica **no documentada en ningún lado** — arreglada de inmediato — y confirmó que el resto del scoping por `gymId`/403-no-404 está implementado de forma consistente en todo el código (`classes`, `plans`, `shifts`, `reservations`, `gyms`, `users`, `reports`, `memberships`).

- **Escalación de privilegios (crítico), corregida el mismo día que se encontró** — cadena de explotación completa y alcanzable sin credenciales previas: `POST /auth/signup-gym` (público) → cuenta `ADMIN` propia con auto-login → `GET /roles` (**sin ningún guard**, exponía el UUID de `SUPER_ADMIN` a cualquiera) → `PUT /users/<propio-id>` con `{roleId: <uuid-super-admin>}` — `UsersService.update()` validaba la reasignación de `gymId` (solo `SUPER_ADMIN` puede tocarlo) pero **nunca validaba `roleId` en absoluto**, así que un `ADMIN` podía autoasignarse `SUPER_ADMIN` sin que nada lo bloqueara → re-login → JWT con `SUPER_ADMIN`, control total de la plataforma. Cubierto con 4 unit tests nuevos en `users.service.spec.ts` (rechazo a `SUPER_ADMIN`, `roleId` inválido, reasignación normal permitida para `ADMIN`, `SUPER_ADMIN` puede asignar cualquier rol sin ir a buscarlo). Verificado en vivo end-to-end contra el servidor y la base de datos reales: la cadena completa de explotación (signup-gym → roles → PUT con roleId de SUPER_ADMIN) ahora devuelve 403 y el usuario queda con su rol original confirmado por SQL; la reasignación legítima (ADMIN promoviendo a un socio de su propio gym a STAFF) sigue funcionando, confirmado por SQL. Fixtures de prueba borrados después.
- **Bug de `timestamp` sin timezone, reaparecido — corregido, 2026-07-21** — la migración `ConvertTimestampsToTimestamptz` (Epic 4, ver arriba) no cubrió cuatro columnas que ya existían antes del fix: `gyms.mercadopago_token_expires_at`, `gyms.mercadopago_oauth_state_expires_at`, `users.reset_password_expires`, `users.email_change_token_expires`. ## Fase 1 del roadmap post-v1.5, implementada 2026-07-24

Del roadmap estratégico (documento publicado, `§06`/`§07`), Fase 1 tenía 4 ítems: Check-in físico, Dashboard MRR/ARR/Churn/LTV, Sucursales, Lista de espera. El usuario decidió secuenciar las 3 independientes primero (este pase) y dejar **Sucursales** como su propia sesión de scoping aparte — toca el modelo de datos central (`Gym`) y tiene implicancias en casi todas las entidades existentes. Pagos (Mercado Pago/Kushki) sigue fuera de cualquier pase, pausado por decisión explícita del usuario hasta que responda soporte de Kushki.

- **Check-in físico** — entidad nueva `CheckIn` (tabla `check_ins`), **no** un cuarto estado de `Reservation`: el comentario propio de `Reservation` (`reservation.entity.ts`) ya documentaba que el modelo de 3 estados (`confirmed`/`cancelled`/`expired`) fue simplificado a propósito para Epic 3 v1.0 y que el check-in físico se dejó para más adelante — mezclarlo como estado de `Reservation` hubiera confundido "la reserva es válida" con "la persona efectivamente vino".
  - Campos: `userId`, `gymId` (denormalizado directo, a diferencia de `Reservation` — un check-in puede no tener clase asociada, ej. gimnasio libre), `reservationId` (nullable), `checkedInAt`/`checkedOutAt` (`timestamptz`).
  - `POST /checkins` (`CLIENT, STAFF, ADMIN, SUPER_ADMIN`): `CLIENT` solo puede hacer check-in de sí mismo (ignora cualquier `userId` del body); `STAFF`/`ADMIN` exige `userId` explícito y valida que el usuario objetivo pertenezca al mismo gym (403, no 404, salvo `SUPER_ADMIN`). Si viene `reservationId`, valida que la reserva exista, pertenezca al usuario resuelto, esté `confirmed`, y que no tenga ya un check-in activo (sin `checkedOutAt` — filtrado con `IsNull()` de TypeORM, no `undefined`, que TypeORM ignora silenciosamente en el `where`).
  - `POST /checkins/:id/checkout`: mismo criterio de permiso que `ReservationsService.cancel` (dueño o `ADMIN`/`SUPER_ADMIN` del gym).
  - `GET /checkins`: mismo patrón de scoping que `ReservationsService.findAll` (`CLIENT`→propios, `SUPER_ADMIN`→todos, resto→del gym).
  - Fuera de alcance de este pase: UI dedicada para check-in de gimnasio libre sin reserva (walk-in) — el backend ya lo soporta (`reservationId` opcional), la UI se agrega después sin tocar el backend de nuevo.
- **Dashboard financiero (MRR/ARR/Churn/LTV)** — `GET /reports/finance` (`ADMIN`, mismo criterio que `/reports/revenue` que tampoco incluye `SUPER_ADMIN` — inconsistencia ya existente, no se tocó), método nuevo `ReportsService.getFinance()` en el mismo archivo que `getOccupancy`/`getRevenue` (reusa `resolveGymId`/`resolveRange` tal cual).
  - **MRR**: `SUM(plan.price_cents)` de `Membership` `active` del gym — snapshot al momento de la consulta, mismo patrón de query-builder que `activeMembersCount` de `getRevenue`, no filtrado por `from`/`to`.
  - **ARR**: `MRR × 12`, sin query propia.
  - **Churn** (aproximación explícita, documentada como tal en el código): `cancelledEnRango / (activosAhora + cancelledEnRango)` — no existe un snapshot histórico de "miembros activos al inicio del período" en este esquema, así que el conteo actual se usa como proxy. No es un cálculo de cohortes real.
  - **LTV**: `(MRR / activeMembersCount) / churnRate`, `null` si `churnRate === 0` (sin churn no hay LTV finito que calcular; el frontend muestra "Sin datos suficientes" en vez de `Infinity`).
  - Requirió una columna nueva: `Membership.cancelledAt` (`timestamptz`, nullable, migración `AddCancelledAtToMemberships`) — hasta ahora no existía forma de saber *cuándo* se canceló una membresía, solo el `status` actual. Se setea en `MembershipsService.finalizeCancellation()` (las dos ramas) y en `syncMembershipFromPreapproval()`, siempre junto con `status: 'cancelled'`.
- **Lista de espera de clases** — entidad nueva `WaitlistEntry` (tabla `waitlist_entries`): `userId`, `classId`, `startAt` (mismo patrón que `Reservation` — una ocurrencia se referencia por `(classId, startAt)`, sin id de ocurrencia materializado), `createdAt` (orden FIFO). Sin campo `status` — se borra la fila al promoverse o al salir, mismo criterio minimalista que el resto del dominio de reservas.
  - `POST /waitlist` (`CLIENT`): mismas validaciones iniciales que `ReservationsService.create` (gym del requester, clase existe y pertenece al gym, membresía activa, `startAt` futuro, `isValidOccurrence`) — pero exige que la ocurrencia **esté llena** (400 explícito si hay cupo: "hay cupo disponible, reservá directamente"). Rechaza duplicados y si el usuario ya tiene una reserva `confirmed` para esa misma ocurrencia.
  - `DELETE /waitlist/:id` (`CLIENT, ADMIN, SUPER_ADMIN`): solo el dueño de la entrada o `ADMIN`/`SUPER_ADMIN` del gym.
  - `GET /waitlist/me` (`CLIENT`): entradas propias ordenadas por `createdAt`.
  - **Promoción automática**: `WaitlistService.tryPromote(classId, startAt)` (privado, no expuesto por controller) — invocado por `ReservationsService.cancel()` después de marcar la reserva `cancelled`. Busca la entrada más antigua para esa ocurrencia, revalida membresía activa + no-solapamiento del candidato (puede haber cambiado desde que se anotó); si pasa, crea la `Reservation` `confirmed` y borra la entrada; si falla la revalidación, borra igual esa entrada (no se puede promover) y sigue con la siguiente — bucle acotado, no recursivo sin límite. Requirió que `ReservationsModule` importe `WaitlistModule`.
  - Notificación de promoción: `MailService.sendWaitlistPromotedEmail()`, mismo estilo fail-open que el resto de `MailService` (sin `MAILTRAP_API_KEY`, solo `logger.warn` y sigue) — primer uso de `MailService` fuera de auth.
- **Verificación**: 201 tests unitarios backend pasando (incluye `CheckInsService`, `WaitlistService` con FIFO/skip-on-failure, `ReportsService.getFinance` con las 4 fórmulas y división por cero en LTV), `tsc`/`build`/`lint` limpios en ambos repos, prueba end-to-end real contra Postgres + servidor locales (clase capacidad 1 → reserva de un socio → segundo socio anotado en lista de espera → cancelación del primero → promoción automática del segundo confirmada por `GET /reservations`/`GET /waitlist/me` → check-in/check-out confirmado por `GET /checkins` → `GET /reports/finance` con valores verificados a mano) y verificación visual en navegador del botón de check-in en `/portal/reservations` (cambia a "Check-out" tras usarse). Fixtures de ambas pruebas borrados de la base de datos después de verificar, sin dejar rastro.

## Convenciones establecidas — no las rompas sin razón

- Todo endpoint nuevo: DTO con `class-validator`, `@ApiOperation`/`@ApiProperty`, y guard explícito (`@UseGuards(JwtAuthGuard, RolesGuard)` + `@Roles(...)`) o deliberadamente público con un comentario de por qué.
- `ValidationPipe` global ya tiene `whitelist` + `forbidNonWhitelisted` — un campo no declarado en el DTO se rechaza solo, no hace falta chequearlo a mano.
- Toda entidad de dominio nueva (a partir de Epic 1) lleva `gymId` (directo o heredado, p. ej. `Reservation` vía `ClassOrResource.gymId`) desde el diseño — la multi-tenancy no se agrega como parche después.
- Tokens de un solo uso (reset de contraseña, verificaciones): usar `TokenService` compartido (`src/common/token/`), no reinventar generación/expiración.
- Nada de `console.log` — `Logger` de `@nestjs/common`.
- Variable de entorno nueva → agregarla a `.env.example` **y** al schema de Joi (`src/config/env.validation.ts`); si no, el arranque no la valida.
- Commits en este repo: **sin** el trailer `Co-Authored-By`.
- Formato de mensajes de commit — [Conventional Commits](https://www.conventionalcommits.org/): `<tipo>(<alcance>): <descripción>`. Ej.: `feat(auth): add biometric login`, `fix(users): resolve pagination bug`. Tipos: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.
